Architecture · v0.4아키텍처 · v0.4

Summarize a running session without breaking the prompt cache. 실행 중인 세션을 요약하면서도 프롬프트 캐시를 깨뜨리지 않는 법.

An LLM agent's KV cache is fragile. Inject a single summary instruction into the live thread and you invalidate every previously cached token — latency spikes, cost balloons, the agent loses focus. This guide explains the prefix-hash invariant, the fan-out-without-fan-in pattern, and how the Session Viewer routes summaries through a forked sidecar thread that shares the prefix and never touches the source.

LLM 에이전트의 KV 캐시는 취약합니다. 활성 스레드에 요약 명령을 하나 주입하는 순간 이전에 캐시된 모든 토큰이 무효화됩니다 — 지연 시간이 급증하고 비용이 폭증하며 에이전트는 초점을 잃습니다. 이 가이드는 접두사 해시 불변성, 팬인 없는 팬아웃 패턴, 그리고 Session Viewer가 어떻게 접두사를 공유하면서 원본 스레드는 절대 건드리지 않는 분기된 사이드카 스레드를 통해 요약을 라우팅하는지 설명합니다.

§01

Caches break the moment you mutate the prefix

접두사를 변경하는 순간 캐시는 깨진다

Modern inference stacks (vLLM, the major hosted APIs) deduplicate prefill work by hashing prompt blocks in order. A single inserted token at any position invalidates every block that follows it.

최신 추론 스택(vLLM, 주요 호스팅 API)은 프롬프트 블록을 순서대로 해싱하여 프리필 작업을 중복 제거합니다. 어느 위치든 토큰 하나가 삽입되면 그 뒤의 모든 블록이 무효화됩니다.

prefix

Transformers compute self-attention against every prior token. As the prompt grows, prefill cost grows quadratically. The fix — KV caching — stores per-layer key/value tensors so subsequent calls that share a prefix can skip prefill entirely.

트랜스포머는 모든 이전 토큰에 대해 자가 주의력을 계산합니다. 프롬프트가 커질수록 프리필 비용이 제곱으로 증가합니다. 그 해법인 KV 캐싱은 계층별 키/값 텐서를 저장하여 접두사를 공유하는 후속 호출이 프리필을 완전히 건너뛸 수 있게 합니다.

The cache is keyed by a chained hash: each block's hash combines the parent block's hash, its own exact tokens, and model parameters. That chain is the entire reason cache hits are fast — and the entire reason any tampering with mid-prompt content is fatal.

캐시는 체인 해시로 키 지정됩니다: 각 블록의 해시는 부모 블록의 해시, 자신의 정확한 토큰, 모델 매개변수를 결합합니다. 이 체인이 캐시 적중이 빠른 이유 전부이며, 프롬프트 중간 내용을 건드리는 것이 치명적인 이유 전부입니다.

§02

Prefix hashing, block by block

블록 단위로 본 접두사 해싱

Inference engines split a prompt into blocks (typically 16 or 32 tokens). Each block produces a deterministic hash from { parent_hash, exact_tokens, model_id }. Cache lookup is exact-prefix matching — there is no fuzzy match.

추론 엔진은 프롬프트를 블록(보통 16 또는 32 토큰)으로 분할합니다. 각 블록은 { 부모 해시, 정확한 토큰, 모델 ID }로부터 결정론적 해시를 생성합니다. 캐시 조회는 정확한 접두사 일치이며, 퍼지 일치는 없습니다.

hash

Order static content first, dynamic content last. The most stable layout looks like: system instructions → tool schemas → reference docs → conversation history (append-only) → current user input.

정적 콘텐츠를 앞에, 동적 콘텐츠를 뒤에 배치하세요. 가장 안정적인 레이아웃: 시스템 명령 → 도구 스키마 → 참조 문서 → 대화 기록(추가 전용) → 현재 사용자 입력.

§03

Provider mechanics differ — and they matter

공급자별 동작은 다르며, 그 차이가 중요하다

OpenAI, Anthropic, and Google each expose different cache primitives. Mismatching your branching strategy to the provider's mechanics is the most common reason a 'should-be-cached' branch silently misses.

OpenAI, Anthropic, Google은 각기 다른 캐시 기본 요소를 노출합니다. 분기 전략을 공급자의 동작과 맞추지 못하는 것이 '캐시되어야 할' 분기가 조용히 누락되는 가장 흔한 이유입니다.

api

Match the breakpoint placement, granularity, and TTL of each provider. The discount tier only kicks in when your routing key lands the request on a node that still holds the warm KV cache.

각 공급자의 중단점 배치, 세분성, TTL을 맞추세요. 할인 티어는 라우팅 키가 따뜻한 KV 캐시를 보유한 노드로 요청을 라우팅할 때만 적용됩니다.

caveat · anthropic
Anthropic's 20-block lookback is the sneakiest gotcha. If your branch adds more than 20 new blocks (turns, tool outputs) past the last breakpoint, the lookback window misses the cached prefix entirely. Park your breakpoints on the last static block — never on volatile message tails. Anthropic의 20-블록 룩백이 가장 교묘한 함정입니다. 분기가 마지막 중단점 이후 20개 이상의 새 블록(턴, 도구 출력)을 추가하면, 룩백 윈도우가 캐시된 접두사에 완전히 도달하지 못합니다. 변동성 있는 메시지 끝이 아닌 마지막 정적 블록에 중단점을 배치하세요.
§04

Why injecting a summary into the live thread fails

활성 스레드에 요약을 주입하면 실패하는 이유

Two compounding failures: the cache breaks (cost), and the agent's reasoning drifts (correctness). One failure mode would be tolerable. Both at once is not.

두 가지 복합 실패: 캐시가 깨지고(비용), 에이전트의 추론이 표류합니다(정확성). 하나만이라면 견딜 수 있지만, 두 가지가 동시에 발생하는 것은 받아들일 수 없습니다.

risk

Performance studies show agent task completion drops by ~39% on average when instructions and exploratory dialogue interleave across multiple turns. Inject 'summarize the above' into a working thread and the model treats it as in-fiction reality — premature conclusions, hallucinated self-evaluations, dropped goals.

성능 연구에 따르면 명령어와 탐색적 대화가 여러 턴에 걸쳐 혼재할 때 에이전트의 작업 완료율이 평균 약 39% 감소합니다. 작업 중인 스레드에 '위 내용을 요약하라'를 주입하면 모델은 이를 픽션 내 현실로 취급합니다 — 조급한 결론, 환각적 자기 평가, 목표 누락.

Worse, the cache miss is silent until billing arrives. A 30k-token conversation that was previously billed at the cached-input rate suddenly pays for full prefill on every turn. That's roughly a 9× cost step-change per request.

더욱 심각한 것은, 캐시 누락은 청구서가 도착하기 전까지 침묵한다는 점입니다. 캐시된 입력 요율로 청구되던 30k 토큰 대화가 갑자기 매 턴마다 전체 프리필 비용을 지불하게 됩니다. 요청당 약 9배의 비용 변화입니다.

§05

The fix: fork the thread, don't poke it

해법: 스레드를 건드리지 말고 분기하라

Take a snapshot of the live thread state at a checkpoint, copy it into a new thread_id, and append your evaluation prompt only there. The forked thread shares the prefix byte-for-byte — every cached block hits.

체크포인트에서 활성 스레드 상태의 스냅샷을 캡처하고, 새 thread_id에 복사한 뒤, 평가 프롬프트를 거기에만 추가하세요. 분기된 스레드는 접두사를 바이트 단위로 공유하므로 캐시된 모든 블록이 적중합니다.

fork

Fan-out without fan-in.팬인 없는 팬아웃. Route the same checkpoint state into both the primary agent node and an evaluation sidecar node in parallel. The sidecar emits its result to telemetry or external storage — never back into the shared graph state. The primary thread's next turn still hits the original prefix cache cleanly. 동일한 체크포인트 상태를 주 에이전트 노드와 평가 사이드카 노드로 병렬 라우팅하세요. 사이드카는 결과를 텔레메트리 또는 외부 저장소로 내보냅니다 — 절대 공유 그래프 상태로 되돌리지 않습니다. 주 스레드의 다음 턴은 여전히 원본 접두사 캐시에 깨끗하게 적중합니다.

§06

Keep history append-only

기록은 추가 전용으로 유지하라

Mutating earlier messages — even shortening tool output — burns the cache from that point forward. Newly summarized context goes at the tail, never in place of older turns.

이전 메시지를 변경하는 것은 — 도구 출력을 줄이는 것조차도 — 그 시점부터의 캐시를 소각시킵니다. 새로 요약된 컨텍스트는 기존 턴을 대체하지 않고 끝에 추가됩니다.

§07

A semantic memory layer keeps summaries out of the prefix

시맨틱 메모리 계층이 요약을 접두사 밖에 둔다

Branched evaluations produce facts, decisions, and trajectory scores. Push them to a vector / graph store (Mem0, vector DB, LangSmith dataset) — never back into the conversation array. Retrieve on demand via tools.

분기된 평가는 사실, 결정, 궤적 점수를 생성합니다. 이를 벡터/그래프 저장소(Mem0, 벡터 DB, LangSmith 데이터셋)로 푸시하세요 — 절대 대화 배열로 되돌리지 마세요. 도구를 통해 필요 시 검색하세요.

mem
# branch produces a fact, push it sideways
result = branch.run()
mem0.add(content=result.summary, metadata={session: "S-104A"})
langsmith.log_evaluation(thread="s_104a_main", score=result.judge)

# next turn on the main thread retrieves only when needed
agent.tools += [get_session_summary]  # dynamic retrieval, prefix stays static
double-caching
The inference layer caches the prefix; the memory layer caches the meaning. The two never collide because the meaning never sits in the prefix. 추론 계층은 접두사를 캐시하고, 메모리 계층은 의미를 캐시합니다. 의미가 접두사에 결코 위치하지 않으므로 둘은 충돌하지 않습니다.
§08

Telemetry: trust but verify the cache hit

텔레메트리: 캐시 적중을 신뢰하되 검증하라

Every provider returns cache_read / cache_write / cache_creation token counts on the response. If your branch's cache_read drops to zero, your prefix invariant has been violated. Treat that as a hard alert.

모든 공급자는 응답에 cache_read / cache_write / cache_creation 토큰 수를 반환합니다. 분기의 cache_read가 0으로 떨어지면, 접두사 불변성이 위반된 것입니다. 이를 강한 경보로 취급하세요.

ops

Pin all branch requests to the same prompt_cache_key as the source thread so the provider's hash-based router lands them on the same backend node — same GPU, same VRAM, near-zero prefill latency. OpenAI's cache-aware routing has been measured at 13–31% TTFT improvement, up to 80% cost savings on heavy workloads.

모든 분기 요청을 원본 스레드와 동일한 prompt_cache_key에 고정하여, 공급자의 해시 기반 라우터가 동일한 백엔드 노드에 — 동일한 GPU, 동일한 VRAM, 거의 0에 가까운 프리필 지연 시간 — 라우팅하도록 하세요. OpenAI의 캐시 인지 라우팅은 TTFT 13–31% 개선, 무거운 워크로드에서 비용 최대 80% 절감으로 측정되었습니다.
